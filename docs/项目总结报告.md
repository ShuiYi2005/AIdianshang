## 四、历史迁移与排障记录（不代表当前运行版本）

本节保留 2026-06 的 `0.15.3` 迁移过程，供排障追溯。当前 Compose 已使用 Dify `1.14.2`，不应按本节命令部署当前项目。

### 挑战 1：Dify 1.x 架构变更导致模型不可用

**问题：** Dify 从 1.0 起将模型供应商管理剥离到独立的 plugin-daemon 服务，该服务的 Docker 镜像 langgenius/dify-plugin-daemon 在国内镜像源全面被墙（403 Forbidden），直连 Docker Hub 超时，导致 Dify 1.x 无法配置任何大模型。

**解决过程：**
1. 调研所有国内镜像源（DaoCloud、腾讯云、1panel.live 等）→ 全部 403 ❌
2. 尝试降级旧版 Dify 0.x → 镜像同样无法拉取 ❌
3. **最终方案：** 找到旧版镜像缓存 langgenius/dify-api:0.15.3，部署 Dify 0.15.3（此版本模型管理内置于 API 服务，不依赖 plugin-daemon）

**涉及技术：** Docker registry mirror、Dify 架构演进、多版本兼容性分析

### 挑战 2：Dify 0.15.3 存储配置兼容

**问题：** 部署 0.15.3 后 API 服务启动失败，报 opendal.ConfigInvalidError: root is not specified。

**根因：** 该版本的存储系统使用 Apache OpenDAL，需显式配置 OPENDAL_SCHEME 和 OPENDAL_FS_ROOT 环境变量。

**解决：** 在 deployment/docker-compose.yml 中为 dify-api 和 dify-worker 添加：
`yaml
OPENDAL_SCHEME: fs
OPENDAL_FS_ROOT: /app/api/storage
`

### 挑战 3：国内网络环境下的 Docker 镜像拉取

**问题：** langgenius 系列镜像（Dify 官方镜像）在国内无法通过任何镜像源拉取。

**关键发现：**
- 基础镜像（postgres、redis、weaviate）正常 → 缓存/镜像源同步了
- langgenius 镜像（包括旧版）全部被墙 → 镜像源未同步该仓库
- 唯一可用的是先前偶然缓存到本地的 :latest 标签

**应对策略：** 使用 0.15.3 版本的 cached 镜像，通过修改 deployment/docker-compose.yml 配置适配旧版本 API。

### 挑战 4：数据库初始化与迁移

**问题：** Dify 的数据库表结构需要手动迁移，初始化时遇到旧版本（1.x）数据库与新部署的 0.15.3 之间的 schema 冲突。

**解决：** 重建 PostgreSQL 数据卷，重新执行 lask db upgrade，从零完成 130+ 张表的迁移。

## 五、历史 Dify 0.15.3 部署步骤（仅追溯）

### 环境准备
- Docker Desktop 29.4.3（Windows）
- WSL2 后端
- 6 容器架构：db / redis / weaviate / dify-api / dify-worker / dify-web

### 部署命令

\\\powershell
# 1. 拉取镜像
docker pull langgenius/dify-api:0.15.3
docker pull langgenius/dify-web:0.15.3

# 2. 启动服务
docker compose -f deployment/docker-compose.yml up -d

# 3. 初始化数据库
docker exec ai20-dify-api-1 flask db upgrade

# 4. 重启 API 服务
docker compose -f deployment/docker-compose.yml restart dify-api
\\\

## 八、项目总结

本项目完整走通了一条 **AI 智能客服系统从 0 到 1 的落地路径**：

1. **方案调研：** 分析了抖店开放平台飞鸽客服系统对接方案，识别出 API Key 资质门槛
2. **技术选型：** 采用 Dify 可选接入、FastEmbed/Weaviate 本地检索与 n8n 编排路线；模型供应商不随仓库配置
3. **应对网络限制：** 保留旧版本适配的历史排障记录，并在当前环境使用可验证的 `1.14.2` Compose 配置
4. **系统部署：** 完成当前本地基础设施、业务模拟服务、`agent-service` 与 React 运营台的 Docker Compose 部署
5. **功能验证：** 当前验证覆盖本地 RAG、模拟订单/库存/物流、训练发布和人工接管闭环；真实电商、模型供应商与生产环境不在本地验收范围内

核心经验：**在受限网络环境下，版本回溯 + 配置适配 是解决 SaaS 平台自托管部署问题的有效策略。**
