# 客服控制台固定壳层与单一主滚动区设计

## 目标

将客服控制台重构为一个占满动态视口的固定应用壳层。页面自身不产生滚动条；在桌面工作台中，聊天消息是默认的主滚动区，右侧商品与证据面板拥有独立滚动区，顶部导航、左侧主导航、会话队列框架、会话头部和底部输入框始终留在屏幕中。

本次重构同时保证训练中心与审计页不会重新引入文档级滚动。它们分别在自身中央内容区域滚动，不改变已有 API、数据模型或业务动作。

## 约束与边界

- 仅修改 `services/support-console` 的 React 结构、样式与前端测试；不修改 `platform/`、后端 API、工作流或数据库。
- 使用 CSS Grid 构建全局壳层和桌面工作台，不以 `position: fixed` 拼接整页。
- `html`、`body` 和 `#root` 均为 `height: 100%` 且 `overflow: hidden`；应用壳层为 `height: 100dvh` 且 `overflow: hidden`。
- 每一个可滚动 Grid 或 Flex 子项均显式设置 `min-height: 0`；横向收缩的子项同时保留 `min-width: 0`。
- 现有的领取、回复、建工单、解决、训练和审计功能、文案、API 调用及错误恢复语义保持不变。

## 信息架构与布局

### 全局壳层

应用由两行 Grid 构成：固定高度的全局顶部导航，以及一个 `minmax(0, 1fr)` 内容行。内容行继续使用两列 Grid：左侧主导航与可收缩主内容。`app-content` 是一个裁剪容器，不允许其子页面将高度撑到文档根节点。

桌面端左侧主导航不是滚动容器；它通过壳层高度自然固定在屏幕内。全局顶部导航同理。它们不会因会话消息、训练素材或审计记录增长而移动。

### 桌面工作台

工作台是三列 Grid：会话队列、会话主栏、商品/证据栏。整个 Grid 使用 `minmax(0, 1fr)` 占用内容区余量并 `overflow: hidden`。

- 队列栏以三行 Grid 组织：标题、筛选器和可滚动队列列表。队列列表在内容溢出时仅在栏内滚动。
- 会话主栏以 Grid 组织为固定的会话头、操作区、可选工单/确认/提示区、`minmax(0, 1fr)` 消息流与底部回复输入框。消息流是默认滚动区；回复输入框是 Grid 最后一行，因此始终贴在中间栏底部而无需 `position: fixed`。
- 商品/证据栏是独立的 `overflow-y: auto` 滚动区。鼠标位于该栏时，只移动其中的客户、商品、风险与证据内容。

会话详情为空时，空状态占用消息流所在的弹性行；不会改变壳层或输入区的总体尺寸。

### 训练中心与审计页

训练中心仍采用主题列表加编辑器的两列 Grid。主题列表与编辑器都约束在壳层可用高度内，编辑器是该页面唯一的主数据滚动区。审计页的表格放入独立的中央滚动区。两页均不得使 `body` 或 `app-content` 滚动。

## 移动端行为

当视口低于桌面断点时，主导航保持紧凑的应用内导航；工作台的会话队列和商品/证据栏不再参与主 Grid 的列宽计算。

- 主聊天栏独占可用宽度和高度。
- 在会话栏顶部提供“会话列表”和“客户信息”两个可访问按钮，用于打开对应的覆盖式抽屉。
- 抽屉相对于应用内容容器覆盖显示，具有遮罩、关闭按钮、`aria-modal` 与清晰的 `aria-expanded` 状态；它们不跟随消息流滚动，也不挤压聊天区域。
- 抽屉自身有受限高度和内部滚动，用于处理很长的队列或证据内容。
- 壳层使用 `100dvh`，回复输入区包含 `env(safe-area-inset-bottom)` 安全区间距。现代移动浏览器的动态视口会在软键盘出现时缩小可用高度，输入框仍处于可视区域。

训练中心的主题列表在移动端也通过同类抽屉呈现，避免其与编辑器纵向堆叠后造成页面级滚动。

## 组件边界

`Workbench` 继续负责队列选择与会话操作，并新增仅表示移动抽屉开闭状态的本地 UI 状态。队列列表和 `EvidencePanel` 的数据接口不变。为保持结构可读性，抽屉容器可提取为仅接受标题、关闭回调与内容的展示组件；它不承载业务请求。

`App` 继续拥有路由和全局导航状态。`TrainingCenter` 只调整其展示层级及移动主题列表开闭状态。不会引入新的后端字段、URL 参数或持久化状态。

## 异常、可用性与兼容性

- 既有 API 加载失败、回复失败和表单输入保留行为不变。
- 抽屉在没有选中会话时照常显示相应空状态。
- 所有新开的抽屉按钮具有可读名称、`aria-expanded` 和可见关闭操作；遮罩点击关闭抽屉。
- 桌面窄至中等宽度时，右侧证据栏由并列列切为覆盖式抽屉，而不是被隐藏且无法访问。
- 不使用整页固定定位；仅允许抽屉作为内容容器中的覆盖层定位，以实现模态层级。

## 验收与验证

1. 在桌面端加载 200 条会话消息并连续滚动消息流，顶栏、左主导航、队列框架、会话输入框和右侧商品卡的屏幕矩形保持不变；仅消息流内容移动。
2. 鼠标或触控板在消息流内滚动时，窗口、队列栏与证据栏不发生滚动；消息流 `scrollTop` 改变。
3. 右侧证据内容超过栏高时，在其上滚动只改变证据栏 `scrollTop`，消息流与窗口位置不变。
4. 在窄屏尺寸下，聊天主栏不因队列或证据而缩窄；打开抽屉后其覆盖主栏，关闭后恢复原位。回复输入框在 `100dvh` 可用高度内且包含安全区间距。
5. 单元测试覆盖移动抽屉的打开/关闭及既有人工回复闭环；构建通过。浏览器验收通过桌面和窄屏视口验证上述滚动边界。
6. Docker Compose 配置、核心 HTTP 验证、前端测试和前端构建均需重新运行并记录成功输出。

## 非目标

- 不接入真实电商渠道、移动端原生键盘 API 或新的外部依赖。
- 不改变客服工作流、训练策略、敏感请求转人工规则或审计持久化。
- 不重绘视觉风格，仅在实现抽屉入口、遮罩和溢出边界时补齐必要样式。
