## 1. 背景与问题

电商商家每天会收到大量重复咨询：商品参数、库存、发货时效、物流进度、催发货、售后、退款和投诉。传统人工客服存在三个主要问题：

- 重复问答占用坐席时间，响应不稳定，夜间与高峰期容易积压。
- 客服难以同时掌握商品、订单、物流、售后政策和客户历史，容易答错或承诺过度。
- 话术、知识和售后处理经验沉淀在个人或零散文档中，无法持续复用与评估。

本产品以“AI 先接待、事实可查询、高风险必转人工、全过程可审计”为核心原则，帮助商家将标准咨询自动化，同时保留人工客服对高风险事项的最终控制权。

## 2. 产品愿景与目标

### 2.1 愿景

让商家能够以可配置、可追溯、可持续优化的方式运行 AI 客服，而不是把客服决策完全交给不可控的模型。

### 2.2 MVP 目标

1. 支持 AI 对文字、图片和视频触发的客服知识进行回复，并提供统一客服工作台。
2. 支持从商品、订单、物流和售后规则中获取可靠上下文，避免模型编造事实。
3. 对退款、赔偿、投诉、身份核验、改地址等风险请求自动转人工。
4. 让运营人员无需改代码即可创建训练主题、上传素材、编辑回复、发布版本和回滚。
5. 在没有真实电商平台 API 时，使用模拟适配器完成端到端演示、测试与验收；接入真实平台时不改变核心业务模型。

### 2.3 成功指标

以下是接入真实渠道后的 30 天观察目标，**不是当前本地环境已测得的经营结果**：

| 指标 | 目标 | 说明 |
| --- | --- | --- |
| 标准咨询自动解决率 | ≥ 60% | 不含主动转人工与客户主动要求人工的会话 |
| AI 首次响应时间 | P95 ≤ 5 秒 | 不含第三方平台长时间不可用场景 |
| 高风险识别召回率 | ≥ 98% | 以人工复核样本与评测集计算 |
| AI 回复可追溯率 | 100% | 每条 AI 回复可关联上下文、知识、工具和策略版本 |
| 人工接管上下文完整率 | 100% | 转人工时必须附带会话摘要、风险原因和相关事实 |
| 训练发布可回滚率 | 100% | 已发布训练版本可一键回滚至任一历史可用版本 |

## 3. 用户与角色

| 角色 | 主要诉求 | 默认权限边界 |
| --- | --- | --- |
| 商家负责人 | 了解客服效率、风险和成本 | 查看经营报表、配置店铺与成员，不直接查看全部敏感字段 |
| 客服主管 | 配置策略、处理升级会话、复盘质量 | 管理队列、工单、训练发布和评测结果 |
| 人工客服 | 接管会话、回复客户、创建工单、添加备注 | 仅访问分配范围内会话；敏感信息按规则脱敏 |
| 运营训练员 | 维护商品知识、售后话术和素材 | 管理训练主题、素材、版本与预览，不处理高风险交易 |
| 审计/管理员 | 审查安全事件和权限 | 查看审计日志；完整个人信息须单独授权 |
| 消费者 | 获得快速、准确、可升级的咨询服务 | 无后台权限；可请求转人工 |

## 4. 产品范围

### 4.1 V1 必须交付

1. **AI 客服工作台**：会话队列、会话详情、消息收发、AI 建议、人工回复、备注、工单、转人工、关闭会话。
2. **AI 训练中心**：训练主题、文本话术、图片/视频素材、适用商品范围、预览、发布、版本记录和回滚。
3. **事实查询与上下文**：订单、商品、物流、客户历史、知识库召回、会话摘要和短期上下文的统一组装。
4. **风险控制**：退款、赔偿、投诉、身份核验、改地址、模型不确定和工具故障时转人工；AI 不执行高风险写操作。
5. **运营治理**：RBAC、数据脱敏、审计日志、模型/知识/工作流版本、评测集、基础指标和成本记录。
6. **模拟闭环**：模拟订单与电商渠道，使所有页面操作和 API 在无真实店铺环境下可测试。
7. **可部署性**：提供 Docker 化本地启动、初始化、验证与离线发布路径。

### 4.2 V1 明确不做

- 自动执行退款、赔偿、改价、改地址、封禁客户等高风险交易操作。
- 未经人工审批的投诉结案或法律、医疗、金融性质承诺。
- 多商家计费、订阅、结算与完整 SaaS 商业化功能。
- 所有电商平台的一次性全量接入；真实渠道按 Adapter 逐个平台上线。
- 将原始平台字段、密钥或未脱敏个人信息直接暴露给模型。

### 4.3 当前实现范围（P0 本地模拟闭环）

当前代码已经实现并可在本机 Docker 环境中验证以下能力：

1. **客服工作台**：待领取、已领取、已解决队列；人工领取、模拟回复、建工单、解决会话；客户、商品、上下文证据与审计记录展示。
2. **训练中心**：训练主题、文本/图片/视频素材上传、回复预览、不可变发布版本与回滚。
3. **本地 RAG**：`BAAI/bge-small-zh-v1.5` 生成 512 维向量；`knowledge/` 下的 `.md` / `.txt` 文档同步到 Business PostgreSQL 与 Weaviate；可在训练中心或 API 发起增量索引。
4. **Agent 策略**：订单、商品、物流等事实通过 `db-simulator` 查询；退款、投诉、赔偿等风险意图转人工；未配置 Dify 时保留本地策略、训练主题、RAG 与转人工降级路径。
5. **工程运行**：Docker Compose 编排 Dify、n8n、PostgreSQL、Redis、Weaviate、模拟 Tool、Agent 服务和 React 运营台。

本阶段的订单、物流、人工发送和渠道消息均是**可审计的模拟链路**；它证明业务模型与接口闭环，不代表已向真实消费者或店铺执行操作。

## 5. 核心业务流程

### 5.1 客户咨询闭环

1. 渠道消息进入系统，依据平台消息 ID 去重并写入会话。
2. 系统识别咨询意图、风险等级和是否需要实时事实。
3. 系统组装上下文：系统策略、当前消息、最近会话、会话摘要、长期记忆、知识召回、订单/商品/物流查询结果与风险规则。
4. AI 在允许范围内生成回复；查询失败或证据不足时明确说明无法确认，不得编造。
5. 对高风险、低置信度、客户要求人工或策略命中的会话，创建转人工记录并进入队列。
6. 每次 AI 回复保存上下文快照、命中知识、工具调用、策略版本和审计事件。

### 5.2 人工接管闭环

1. 客服在队列中领取或被分配会话。
2. 工作台展示客户画像、订单/物流摘要、AI 对话摘要、风险原因、历史备注和待办工单。
3. 客服发送人工回复、添加内部备注、创建/更新工单，或将会话转交其他坐席。
4. 完成后关闭会话；关键状态变更和操作人进入审计日志。

### 5.3 训练与发布闭环

1. 运营人员创建训练主题，例如“售后问题”“发货时效”或“商品材质说明”。
2. 为主题录入触发描述、回复话术、图片/视频素材、适用商品与风险限制。
3. 系统提供预览和测试对话，标明素材命中与预计回复。
4. 训练员发布不可变版本；发布前通过必填校验和敏感内容检查。
5. 发布后按商品范围生效；发现问题时可回滚到历史版本，并保留发布与回滚审计。

## 6. 功能需求

### 6.1 客服工作台

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| CS-01 | 展示待处理、处理中、已完成、已关闭会话队列 | 队列筛选不影响其他坐席；状态变更即时反映 |
| CS-02 | 展示客户消息、AI 消息、人工消息和系统事件 | 消息按时间排序，来源与发送状态可区分 |
| CS-03 | 固定输入区支持文本、常用话术、附件入口与发送 | 连续滚动历史消息时输入区始终可见 |
| CS-04 | 右侧展示客户、商品、订单、物流、工单和上下文摘要 | 右栏可独立滚动，不影响消息区滚动 |
| CS-05 | 支持备注、领取、转交、转人工、创建工单和关闭会话 | 每项操作写审计日志并返回明确结果 |
| CS-06 | 显示 AI 建议回复及其证据来源 | 人工可编辑后发送；高风险建议不得直接自动发送 |

### 6.2 AI 回复与风险策略

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| AI-01 | 支持 FAQ、商品说明、售后政策等知识问答 | 回复可关联知识版本与引用摘要 |
| AI-02 | 对订单、物流、商品库存等问题调用事实查询工具 | 工具结果优先于模型记忆；工具失败时不虚构答案 |
| AI-03 | 支持图片、视频触发的训练素材回复 | 命中条件、素材版本和回复内容均可追溯 |
| AI-04 | 识别退款、赔偿、投诉、身份核验、改地址等风险意图 | 命中后创建人工接管，不执行高风险写操作 |
| AI-05 | 支持客户主动要求人工 | 一次操作进入人工队列，并生成交接摘要 |
| AI-06 | 支持工具超时、知识无命中和模型异常降级 | 对客户给出可理解说明，必要时转人工 |

### 6.3 训练中心与知识管理

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| TR-01 | 创建、编辑、停用训练主题 | 主题包含名称、触发描述、业务分类、适用范围和状态 |
| TR-02 | 上传和管理文本、图片、视频素材 | 素材有类型、来源、预览、大小限制和审核状态 |
| TR-03 | 配置 AI 回复、适用商品和优先级 | 同一咨询多个主题命中时可按优先级和范围决策 |
| TR-04 | 发布不可变训练版本 | 已发布版本不能原地修改，只能新建版本 |
| TR-05 | 预览、灰度和回滚 | 可指定商品范围测试；回滚保留原因与操作者 |
| TR-06 | 知识文档版本化和检索 | 文档来源、版本、切片与检索结果可审计 |

### 6.4 订单、商品与渠道适配

1. 核心业务模型使用统一的客户、商品、订单、物流、售后和会话对象，不依赖抖音、淘宝、京东等平台私有字段。
2. 每个平台通过独立 Adapter 处理鉴权、签名、字段映射、状态映射、限流、重试、Webhook 解析和幂等。
3. 原始平台载荷只保存到受控审计或 Webhook 事件中，不直接进入模型上下文。
4. V1 使用模拟 Adapter；真实平台上线前必须补齐字段映射、权限范围、状态枚举、错误码、速率限制和回调对账测试。

### 6.5 权限、安全与审计

1. 使用 RBAC 加数据访问策略；高风险权限如退款处理、完整个人信息查看、审计导出必须单独授权。
2. 默认脱敏手机号、地址和其他个人信息；AI 默认仅获得完成任务所需的脱敏数据。
3. 每条 AI 回复、工具调用、人工操作、训练发布、权限变更和数据导出必须记录审计事件。
4. 所有外部写操作必须带幂等键；Webhook 必须按提供方事件 ID 去重。
5. 密钥只能出现在被忽略的运行时环境文件或密钥管理系统，不得提交到代码仓库、训练素材或日志。

## 7. 数据与状态要求

### 7.1 数据优先级

| 数据类型 | 权威来源 | 使用规则 |
| --- | --- | --- |
| 实时订单、库存、物流、售后状态 | 业务库或平台 API | 必须优先查询，不可由模型记忆替代 |
| 商品说明、FAQ、售后政策 | 已发布知识版本 | 可检索引用，必须标识版本 |
| 客户偏好、会话摘要、风险标记 | 长期记忆 | 仅作辅助，不覆盖实时事实 |
| 最近会话与工具缓存 | Redis 短期记忆 | 必须设置 TTL，可丢失并可重建 |
| 回复上下文和操作证据 | 上下文快照与审计库 | 用于追溯、质检与争议处理 |

### 7.2 会话状态

`待处理 → AI 处理中 → 待人工 → 人工处理中 → 已解决 / 已关闭`

状态转换必须记录来源、操作者、时间、原因和关联工单。人工客服可关闭会话；系统不得因 AI 单次回复自动关闭高风险会话。

## 8. 非功能需求

| 维度 | 要求 |
| --- | --- |
| 可用性 | 核心服务必须提供健康检查；关键依赖不可用时可识别与降级 |
| 性能 | 工作台主操作响应 P95 ≤ 2 秒；AI 首次响应 P95 ≤ 5 秒（不含外部平台长故障） |
| 可观测性 | Trace ID 贯穿渠道、工作流、工具、AI 回复与审计；监控成功率、超时率、转人工率、RAG 命中率和成本 |
| 可维护性 | Prompt、工作流、适配器契约、知识版本与训练版本均文件化或版本化管理 |
| 可靠性 | Webhook、消息、工单和写操作需去重、重试、幂等和失败告警 |
| 部署 | 新环境可通过 Docker Compose 启动；提供本地环境生成、验证、离线发布和恢复说明 |
| 移动端 | 窄屏下会话为主视图，左右信息栏使用抽屉或覆盖层，输入框不被键盘遮挡 |

## 9. MVP 验收标准

以下场景全部通过后，MVP 才可进入真实店铺接入准备阶段：

1. 客户咨询商品、订单、物流时，AI 能正确调用模拟数据并回复；无法确认时明确说明限制。
2. 客户发送训练命中的图片或视频素材时，系统能返回对应内容并记录素材版本。
3. 退款、赔偿、投诉、身份核验和改地址请求不自动执行，均转入人工队列并附带上下文。
4. 人工客服可在工作台完成领取、回复、备注、建单、转交和关闭；每一步有审计记录。
5. 训练主题可创建、预览、发布、查看历史版本和回滚；发布前后行为可验证。
6. 切换 200 条会话消息滚动时，桌面端左栏、顶栏、输入框和右侧商品卡位置保持稳定；右栏独立滚动。
7. 无真实电商后台时，模拟 Adapter 能支撑上述端到端测试；接入真实平台时 Adapter 契约测试必须通过。
8. 安全扫描不得发现提交的真实密钥；默认角色不可查看完整敏感个人信息。
9. Docker Compose 配置、服务健康检查、训练 API、RAG、审计和评测脚本均通过；Dify Chatflow 与真实模型仅在配置凭据、发布应用后作为单独验收项，不得由本地降级结果替代。

## 10. 分阶段发布

| 阶段 | 目标 | 进入条件 |
| --- | --- | --- |
| P0：模拟闭环 | 完成工作台、训练、AI 回复、风险转人工和模拟数据 | **核心实现完成**：本地 Docker 可运行；仍保留 GitHub 新克隆静态验收脚本待排查项 |
| P1：试运营 | 选定一个商家与一个低风险渠道灰度使用 | 评测集、监控、审计、权限和回滚机制稳定 |
| P2：真实渠道接入 | 接入一个真实电商平台 Adapter | 鉴权、字段映射、限流、Webhook 对账和异常演练通过 |
| P3：规模化 | 支持多店铺、更多渠道与运营分析 | 多租户隔离收紧、容量评估和运营流程成熟 |

## 11. 关键产品决策

1. **AI 负责建议与标准答复，人工负责高风险决策。** 这是对交易风险、消费者体验和可追责性的默认保护。
2. **实时事实与知识、记忆分层。** 订单和物流必须查事实来源；知识库和记忆不能覆盖实时状态。
3. **先模拟、后接真实渠道。** 模拟环境用于验证产品闭环；真实平台差异由 Adapter 隔离，避免业务核心被平台字段绑死。
4. **发布即版本化。** Prompt、知识、训练素材、工作流和策略都必须可追溯、灰度和回滚。
5. **默认最小权限与脱敏。** 只有在明确授权与业务必要时，人才或系统才能访问完整个人信息。

## 12. 真实平台接入前置条件

真实店铺接入不是 MVP 的阻塞项，但在 P2 前必须由商家确认并提供：

- 目标平台、店铺主体与授权方式。
- 可读取和可写入的 API 权限范围。
- 订单、售后、退款、物流状态的业务口径。
- 人工客服值班、升级 SLA、退款与赔偿审批规则。
- 数据保留期限、隐私合规要求和客户告知文本。

在上述条件未满足前，系统只能以模拟渠道展示“已发送”状态，不能宣称已向真实消费者或真实电商后台执行操作。

## 14. 当前验收结论与下一步

当前项目满足“本地模拟电商 AI 客服 MVP”的核心演示与工程验证目标，适合用于功能演示、代码评审、内部测试和真实渠道接入前的 UAT 准备。它尚不满足生产就绪或正式交付条件。

下一阶段按优先级为：

1. 修复 GitHub 新克隆静态验收脚本，并冻结一份可复现的 P0 验收报告。
2. 接入身份认证与运营权限，至少保护索引、训练发布、审计导出等管理操作。
3. 选择一个真实电商渠道，完成 Adapter、只读订单/物流查询、Webhook 与灰度 UAT。
4. 在受控环境配置模型供应商与已发布 Dify Chatflow，完成质量、成本、失败降级和审计验收。
5. 完成生产运维、备份恢复、安全和业务签收后，再评估“生产就绪”与“正式交付”。
